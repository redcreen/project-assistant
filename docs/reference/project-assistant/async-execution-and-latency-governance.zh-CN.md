# 异步执行与时延治理

[English](async-execution-and-latency-governance.md) | [中文](async-execution-and-latency-governance.zh-CN.md)

## 目的

这份说明把用户反馈的速度问题升级成一个正式专项，并记录当前已经确认的方向：

`目标架构直接走 daemon-first PTL 调度器，但第一版先做 write-safe 的快升级版，先把写代码速度提起来，再逐项验证旧功能。`

它要回答的是：

`哪些任务必须同步，哪些任务应该默认异步，以及长期应该由什么架构来承载这套策略？`

## 北极星对齐

这次专项不能只围绕“更快的执行器”来设计。

它必须继续服务 `project-assistant` 的原始目标：

`帮助人类更规范、更专业地写项目代码，并把注意力放在需求、方向、边界与取舍上，而不是陷在过程细节和过程控制里。`

因此这里的时延治理不是为了增加一层更重的调度系统，而是为了把不该由人盯着的过程负担搬走。

## 模板化预生成文档作为加速器

专项不应只考虑任务执行异步化，也要考虑文档生成提速。

当前已确认一个方向：

- 可以增加模板化、预生成的文档骨架
- 用它更快落下 `brief / architecture / roadmap / test-plan / development-plan / status` 的第一版
- 再把人类注意力放在真正要讨论和裁决的内容上

边界也要明确：

- 模板是加速器，不是新的流程负担
- 预生成内容应尽量可裁剪、可增量刷新
- 不应为了“更完整的模板”牺牲首屏速度和主线流畅度

## 问题定义

当前问题已经不是“某几个脚本偏慢”。

用户明确表达的是：

- 仍然有太多工作以同步方式运行，持续打断主编码线
- 整个 skill 对日常开发来说太重
- 如果继续这样慢下去，这个 skill 会直接失去使用意愿，甚至被卸载
- PTL 应该更像前置讲解、同步审查、checkpoint 反馈，而不是持续 inline 干扰

所以这已经是一个产品留存问题，而不是一个局部优化问题。

## 硬约束

| 约束 | 含义 |
| --- | --- |
| 能异步就默认异步 | 只要任务不必阻塞当前主写入线，就应进入后台 |
| 同步任务必须更短 | 目标不只是“1 分钟内”，而是尽量更严格地压缩 inline 阻塞 |
| 长任务必须有 ETA | 用户要知道是该等、继续写代码，还是放着后台跑 |
| 主写入线必须受保护 | 后台任务不能随机打断或抢占当前编码线程 |
| PTL 要监督，不要悬停 | PTL 应通过前置 brief、checkpoint review、事件触发升级来工作 |

## 已确认方向

目标架构现在定为：

`daemon-first PTL 调度器`

但实现策略不是“一口气做全功能”，而是：

`先做 write-safe 的快升级版`

这意味着：

- 先尽快把适合后台化的工作交给常驻 daemon
- 先解决“写代码速度慢、主线总被打断”的问题
- 第一版先不让 daemon 自动写业务代码
- 等 daemon 基线落地后，再把旧功能项按家族逐个验证

同时加一条方法论约束：

- daemon 和模板系统都只是加速层，不能反过来把人重新拖回过程控制

## 方案重新评估

| 选项 | 形态 | 当前判断 |
| --- | --- | --- |
| A. 保持当前同步模型，只继续做局部脚本优化 | 只做局部提速 | 放弃；不足以拯救可用性 |
| B. 先做后台优先执行模型，但暂时不上常驻 daemon | 规则先行，运行时后补 | 可以作为过渡，但已不再是目标架构 |
| C. 直接上常驻 daemon 式 PTL 调度器 | 常驻运行时、任务队列、ETA、checkpoint review、后台执行 | 选定为目标架构 |

## 目标架构

### 1. 前台 / 后台分层

| 通道 | 放什么 | 首版规则 |
| --- | --- | --- |
| 受保护的前台主写入线 | 当前代码主任务、需要用户立即裁决的判断、极短安全检查 | 同时只保留 1 条主写入线 |
| 后台 checkpoint 线 | validator、snapshot、progress/handoff 生成、docs sync、control-surface refresh、benchmark / audit | 默认异步 |
| 后台长任务线 | 长审计、release-ready 检查、打包、全仓整改分析 | 异步 + ETA |

### 2. 同步预算

建议首版快升级按更贴近真实开发体验的预算执行：

| 档位 | 规则 |
| --- | --- |
| `0-10s` | 可静默 inline |
| `10-30s` | 可 inline，但应给出预期 |
| `>30s` | 默认后台，除非用户明确要等待 |
| `>60s` | 默认不允许阻塞 inline |

### 3. PTL 行为

daemon 驱动的 PTL 应该按这种节奏工作：

1. 开始前先给 worker brief
2. 让 worker 安静推进
3. 监听队列事件、失败、完成和 checkpoint
4. 到 checkpoint 再统一复核
5. 只有边界、优先级、兼容性、成本或策略变化时才升级

## 为什么要 daemon-first

| 收益 | 为什么重要 |
| --- | --- |
| 能先把写代码速度拉起来 | 支撑类任务不再持续阻塞主线 |
| ETA 更自然 | daemon 天然适合维护队列状态、耗时和时长档位 |
| PTL 更像真实工作方式 | 监督变成 checkpoint / 事件逻辑，而不是持续 inline 微管理 |
| 旧功能验证更容易分批做 | daemon 基线先落地，然后按家族逐个回归 |
| 模板与生成动作更容易后台化 | 文档骨架生成和一致性刷新可以离开主编码线执行 |

## 主要风险

| 风险 | 为什么危险 | 首轮缓解 |
| --- | --- | --- |
| 后台任务碰到错误写入范围 | 会破坏主写入线 | 首版明确禁止 daemon 自动改业务代码 |
| daemon 状态和 durable 真相漂移 | 用户会看到不一致状态 | durable 真相继续留在 `.codex/*`，daemon 队列状态单独存放 |
| daemon 变黑箱 | 用户失去信任 | 必须有任务状态、ETA、cancel、retry 和明确 ownership |
| rollout 过大 | 快升级版迟迟发不出来 | 先做窄 MVP，再逐项验证旧功能 |

## 快升级版

第一版要优先服务“马上把开发体验拉回来”，而不是一开始就追求完整。

### 首版范围

- 本地常驻 PTL daemon
- 任务队列，至少支持 `queued / running / waiting-checkpoint / completed / failed / cancelled`
- ETA 或时长档位
- 1 条受保护的前台主写入线
- 只后台化安全支撑任务家族
- 模板化、预生成文档骨架的快路径
- checkpoint 式结果回报
- kill switch 和退回当前非 daemon 路径的 fallback

### 首版明确不做

- 自动后台写业务代码
- 多 worker 代码写入调度
- 跨宿主 / 多桌面 Codex 编排
- 激进地自动重排主写入线
- 为了模板完整度而引入更重的前台阻塞

## 快升级后的验证策略

不要等所有旧功能都 daemon-native 后才发第一版。

应该先发快升级版，然后按顺序逐个验证旧功能家族：

1. daemon 队列本身：状态、ETA、cancel、retry、checkpoint 回报
2. `continue / progress / handoff`
3. `bootstrap / retrofit / docs-retrofit`
4. control-surface 和 docs validators
5. development-log 与 release 邻近流程
6. 代表性仓库上的更广泛 rollout 验证

## 建议 rollout

| 阶段 | 目标 |
| --- | --- |
| Phase 1 | 锁定 daemon-first 架构与 write-safe 快升级版范围 |
| Phase 2 | 先发 PTL daemon 快升级版，把活跃编码速度拉起来 |
| Phase 3 | 在 daemon 基线之上，按功能家族逐项验证旧能力 |
| Phase 4 | 基线稳定后，再逐步扩大后台异步覆盖面 |
