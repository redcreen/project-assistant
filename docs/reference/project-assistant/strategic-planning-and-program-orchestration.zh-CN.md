# 业务规划与程序编排方向

[English](strategic-planning-and-program-orchestration.md) | [中文](strategic-planning-and-program-orchestration.zh-CN.md)

## 当前状态

这份文档记录 `project-assistant` 在当前工作模型之上已经确认的下一层方向：

- 战略评估
- 程序编排
- 受监督的长期自动交付

这个方向现在已经进入正式路线图。`M10` 战略评估层、`M11` 程序编排层、`M12` 受监督的长期自动交付层、`M13 PTL 监督环`、`M14 worker 接续与回流` 和 `M16 统一硬入口与工具前门` 都已经完成；当前进入 post-M16 rollout 验证，用真实 repo 证据判断 `M15 选择性多执行器调度` 是否真的需要；`M8 / M9` 继续作为这条方向下的 supporting backlog。

这里默认把负责这三层长期推进与升级判断的角色统一叫作：

- 项目技术负责人（PTL）

## 为什么需要这一层

`project-assistant` 现在已经把很多执行侧能力做得比较完整：

- 控制面
- 长任务执行线
- 架构监督
- `progress / continue / handoff`
- 文档整改与 Markdown 治理
- 开发日志
- 验证与发布门禁

但它还没有完全覆盖这个更高层的问题：

`项目接下来应该怎么走，以及当人类不持续盯着时，谁来确保多条长期切片仍然沿着正确方向推进？`

这个空缺会在下面几类场景里反复出现：

| 场景 | 为什么现有层还不够 |
| --- | --- |
| 项目不断暴露根因漂移和专项收口需求 | 执行层可以继续推进，但没有明确的战略层决定何时插入治理专项或架构专项 |
| 旧 milestone 已不再符合现实 | 当前层能看见漂移，但还不会正式提出 roadmap 改造建议 |
| 一个项目已经不只是一条 slice 或一个执行器 | 当前 execution board 只能局部收口，缺少跨 workstream 的编排层 |
| 人类更想只管业务方向，不想持续盯交付过程 | 现有流程已经减少 babysitting，但还没有一个更高层的 AI 监督角色长期盯到真正需要业务裁决为止 |

## 角色模型

| 层 | 主要 Owner | 责任 |
| --- | --- | --- |
| 业务方向层 | 人类 | 提需求、定优先级、定方向、做重大取舍 |
| 战略规划层 | 项目技术负责人（PTL） | 判断项目后续怎么走、是否要插治理/架构专项、是否要调整 roadmap 结构、项目定位是否要提升 |
| 程序编排层 | 项目技术负责人（PTL） | 管多个 workstreams、切片、依赖和执行器的长期推进 |
| 执行层 | AI 交付执行器 | 写代码、改测试、补文档、跑验证、更新控制面 |
| 治理与恢复层 | 当前 `project-assistant` 能力 | 维护当前真相、门禁、进展、开发日志、文档与恢复链路 |

## 核心原则

人类仍然应该负责：

- 业务方向
- 产品定位
- 重大兼容性决定
- 显著的成本 / 时间取舍
- 需求变化本身

AI 系统则应逐步承担：

- 战略评估建议
- 在已批准方向内的长期编排
- 持续推进多个切片
- 只在真正需要人类判断时升级

PTL 在这里的角色不是产品 owner，也不是组织级 CTO。它更像一个项目级技术主责人：

| PTL 负责什么 | PTL 不负责什么 |
| --- | --- |
| 战略评估、程序编排、长期交付监督、升级判断 | 业务方向变化、产品优先级、重大兼容性和成本取舍 |

## 里程碑方向

### M10：战略评估层

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 判断项目后续应该怎么走、是否应插入治理/架构专项、既有 milestone 或项目定位是否需要调整 |
| 产物 | durable 的 PTL 战略判断与可 review 的战略控制面，而不是自动修改业务方向 |
| 人类角色 | review 并批准或否决策略变更 |
| 退出条件 | 战略建议变成显式、可审阅、可追溯到 repo 证据的输出 |

### M11：程序编排层

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 不再只管理一条执行线，而是管理多个 workstreams、切片和执行器 |
| 产物 | durable 的 PTL program board，用来表达活跃流、先后顺序、可并行边界和调度检查点 |
| 人类角色 | 批准编排边界与升级规则 |
| 退出条件 | 系统可以在不频繁依赖人工“继续”的前提下，持续推进几条相关 slice |

| 当前边界 | 说明 |
| --- | --- |
| 现在已经做成的 | 一个 Codex 内的 durable 编排真相层：它知道有哪些 workstreams、哪些能并行安全推进、哪些必须串行，以及当前 PTL / delivery worker / docs-and-release 各自负责什么 |
| 现在还没做成的 | 通过 `项目助手` 一句话自动拉起多个桌面 Codex 会话、分发任务并回收结果 |
| 后续是否要做 | 只有当 rollout 证据表明“单 Codex 编排真相层”已经不够，才单独立项做多执行器调度层 |

### M12：受监督的长期自动交付

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 让 AI 交付系统能持续推进长期工作，直到完成或遇到真正的业务裁决点 |
| 产物 | 更长时间运行的 PTL 受监督执行流，带周期性检查点、自动升级和 durable 恢复真相 |
| 人类角色 | 只在方向变化或取舍需要批准时介入 |
| 退出条件 | 人类主要负责方向与决策，而不是持续输入“继续” |

## Durable 控制面

| 控制面 | 作用 |
| --- | --- |
| `.codex/strategy.md` | 当前战略判断、专项插入建议、roadmap 调整建议、需要人类裁决的点 |
| `.codex/program-board.md` | 活跃 workstreams、编排状态、串并行边界、执行器分工和调度检查点；当前先承载单 Codex 的总协调真相 |
| `.codex/delivery-supervision.md` | checkpoint 节奏、自动继续边界、升级时机、执行器监督循环和 backlog 回流规则 |

这两者应该和现有控制面并列，而不是替代 `plan / status`。

## 行为型 Backlog 候选

| 主题 | 含义 | 当前定位 |
| --- | --- | --- |
| 问题驱动收口环 | 当 durable 问题被识别后，系统未来应自动执行：记录当前问题、解决思路和方案；把关键结论写入 devlog 与 architecture；同步 roadmap / development plan；再一口气进入长任务实现 | supporting backlog / todo |

## 自动化边界

| 系统可以自动做 | 系统必须升级给人类 |
| --- | --- |
| 提出治理或架构专项建议 | 改产品方向 |
| 提出 roadmap 调整建议 | 改外部承诺 |
| 在已批准方向内编排多个切片 | 改兼容性承诺 |
| 在检查点之间持续推进长期工作 | 接受显著的成本 / 时间变化 |
| 在已批准范围内重排技术顺序 | 需求变化不经 review 就直接吸收 |

## 落地顺序

1. 先定义战略评估文档结构和 review 合约
2. 让战略输出必须基于 repo 证据，而不是只给空泛建议
3. 再引入轻量 program board
4. 等 program board 和升级规则稳定后，再允许跨多个切片做更强编排
5. 最后才考虑更长时间的受监督自动推进

## 当前批准边界

方向已经批准，但它的自动化边界仍然明确受限：

1. `project-assistant` 可以提出 roadmap 重排、治理专项和架构专项建议。
2. 它不能在没有人类批准的情况下自动改变业务方向、兼容性承诺或对外定位。
3. `M8` 的 locale-aware internal output 与 `M9` 的 continue 压缩，现在都转成 `M10` 下的 supporting backlog，而不是继续占用主线里程碑。
4. `M11` 已经把 durable program board、编排边界和维护者展示收口完成；`M12` 也已经把长期监督交付层收口成 durable `delivery-supervision` 面。
5. `M13` 与 `M14` 已经收口成 durable 的 `PTL supervision` 与 `worker handoff` 控制面、门禁、进展与交接。
6. `M15` 仍是受证据约束的后续层：如果任务没有不相交写入边界，或跨 repo rollout 没证明单 Codex PTL 模式已成瓶颈，就不应进入多执行器调度。
7. `M16` 已把 `continue / progress / handoff` 收成统一前门、版本 preflight、结构化第一屏和 durable `entry-routing` contract；桌面宿主级强绑定仍是后续桥接问题，不应被提前宣称成已完成。
8. “问题驱动收口环”已经是稳定出现的请求模式，后续应评估把它升级成默认行为，而不是继续依赖用户逐条提醒“写日志、写架构、改 roadmap / plan、然后实现”。

## Post-M12 正式主线

### M13：PTL 监督环

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 让项目技术负责人（PTL）不再只在聊天里被动出现，而是通过周期性 / 事件驱动的监督环持续盯住项目推进 |
| 产物 | durable 的 PTL supervision loop：它知道什么时候巡检、什么时候继续、什么时候重排、什么时候升级 |
| 人类角色 | 仍然只在业务方向变化、重大取舍或兼容性边界变化时介入 |
| 当前状态 | 已完成，并已沉淀为 `.codex/ptl-supervision.md`、对应 sync / validate 脚本，以及 `progress / continue / handoff` 里的 PTL 监督摘要 |
| 退出条件 | worker 停下后，项目不会一起停住；PTL 能继续推动项目到下一个 checkpoint 或裁决点 |

### M14：worker 接续与回流

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 让 worker 在 checkpoint、超时、失败或交接后，剩余工作仍能被 PTL 恢复、转交、回队列，或者明确升级 |
| 人话解释 | `worker 停了，项目不能跟着停` |
| 产物 | durable 的 handoff / re-entry contract：它表达剩余任务、恢复入口、回流规则和升级条件 |
| 人类角色 | 只在接续过程中暴露业务方向、兼容性或成本边界变化时介入 |
| 当前状态 | 已完成，并已沉淀为 `.codex/worker-handoff.md`、对应 sync / validate 脚本，以及 `progress / continue / handoff` 里的 handoff 摘要 |
| 退出条件 | worker 不再是“停下就丢活”；项目能通过 PTL 把未完成工作接住并继续 |

### M15：选择性多执行器调度

| 项目 | 当前含义 |
| --- | --- |
| 目标 | 只对安全并行任务开放真正的多执行器调度 |
| 产物 | 明确的 executor assignment、write-scope contract、结果回收口和冲突门禁 |
| 人类角色 | 审批明显扩大成本、复杂度或外部影响的调度层扩张 |
| 当前状态 | 仍保持 later / evidence-gated；只有在更多 repo 上证明单 Codex PTL 模式真的成为瓶颈时，才进入实现 |
| 退出条件 | 只有边界清楚、冲突可控的任务才进入多执行器；高耦合任务继续保持单主写入线 |
