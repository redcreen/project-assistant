# 记录系统性时延问题与异步执行硬约束

- 日期：2026-04-13
- 状态：active

## 问题

用户明确指出当前工作流整体过慢，而且问题不只在单个脚本耗时，更在于太多工作仍以同步方式插入主线，导致开发阶段持续被打断。用户还要求：能异步的尽量异步；同步任务尽量控制在 1 分钟内；长任务必须给出明确预期；PTL 模型应更像前置讲解、同步审查与定期检查反馈，而不是一直干扰正在写代码的 worker。

## 思考

此前已经做过一轮 repo 内部快修，例如去掉重复 bootstrap pass 和补统一前门，但这只能改善局部。当前问题已经升级成执行模型设计问题：需要先把任务分类、同步预算、异步默认规则、长任务预期契约和 PTL 监督节奏设计清楚，否则继续做零散优化会反复回到“还是太慢”的反馈。

## 解决方案

先不直接开做实现，而是把问题正式记录下来，并建立一个专项设计底稿 `async-execution-and-latency-governance`。该专项先回答哪些任务必须同步、哪些任务应默认后台化、同步预算怎么定、长任务如何给 ETA，以及 PTL 如何改成 checkpoint / 事件驱动监督。等方案评审通过后，再开始真正的异步执行治理实现。

## 验证
